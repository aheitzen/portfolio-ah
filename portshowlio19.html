Portshowlio 19

Push. Pull. Adapt.

Each Spring, the graduating class of SCCA invites friends, family, and the Seattle design community to a two day celebration. We transform our classrooms into an interactive design space where we can all come together to celebrate the work from the past two years.

The students are split into teams such as branding, environmental, and web. This year, I had the privilege to lead the web team. We set out to create a responsive website that will host our work for the show, allow students to successfully upload their work to the site, and act as an archive of our work at SCCA.

The Opportunity

The primary opportunity for the website is to create a place where show attendees can browse student work before the show, obtain general information about the show, and reference designers work that stood out to them after the show.

The secondary opportunity for the site is to create an archive where our work at SCCA can be memorialized and used as inspiration and marketing material for poterinal and future SCCA students.

Project Goals

  • Lead the web development team effectively and efficiently
  • Strengthen communication, leadership, and project management skills
  • Create clear documentation that allows students to submit work to the website with confidence
  • Build a responsive website using PHP/WordPress for the 2019 Portshowlio

The Process

Project management

The first step in the project was to define a system where we could all communicate effectively with each other. We utilized Slack, Trello, printed calendars, and weekly meetings to stay up-to-date on progress as well as pain points. I created the Trello board, calendar, pinterest boards, and lead the weekly meetings.It was also my responsibility to distill key takeaways from lead meetings, smaller meetings with the leads of each team.

Divide and Conquer

Now that the team had means to communicate effectively it was time to divide the work. I decided to divide the team of six into two groups of three, three designers and three developers. Feedback would cross that distinction as we all have skills in design, but modularizing the group into smaller teams proved effective in dividing and expediting work.

The design team began to work on lo-fi prototypes of the web pages, while the development team sourced for innovative web technologies around movement. I began to plan out the site architecture and familiarize myself with the WordPress environment.

Form Building

During the above mentioned process, I began building a submission form for the website. Using a WordPress plugin called Advanced Custom Fields, I was able to connect form data to the webpage. While building the form, I was limited by WordPress UI, which can be confusing and is not user-friendly. I put emphasis on usability and annotated every input field, and used language that was easily digestible.

In addition to providing scaffolding via captions, I also created a form submission document where all information relating to saving images for web and uploading content to the site successfully could be found. Throughout this process, I learned that clear communication and documentation can solve many individuals common pain points.

Responsive web development

Building the website involved many revisions and pivots. Initially we believed we could use CSS grid and flexbox as the primary grid system when populating the pages with student submissions. We quickly ran into trouble when introducing logic into that paradigm. CSS grid and flexbox worked very well on certain pages, where we had more control over student submissions, dimensions, and overall layout, but on the homepage specifically the logic and layout were not playing nicely.

These pain points ultimately lead to simplifying two pages of the site, the homepage and single student post page. Instead of working with the full funky power of CSS grid, I went back to basics and simplified layout and solidified image upload dimensions.

Once I had content from the form successfully rendering in the browser, I began to code using html and css to perfect the layout, and added JavaScript where necessary, such as the search and filter functionality, and other small interaction patterns like scrolling to a named anchor point, and hover functionality in the navbar. Finding the PHP syntax and attributes I needed, I referenced the WordPress codex, and referenced Stack Overflow for other WordPress/PHP questions.

Quality control

Now that the website was successfully being populated by student work from form submissions, it was time to make sure the uploads were successful. Due to various problems with the ACF plugin, I was unable to put the correct constraints on image uploads, leading to inconsistent images sizes. Other inconsistencies from data included forgetting to fill in certain lines, and misspellings. Sean and Ian ran quality control on the site, and documented these minor mistakes. They then approached the individuals to close the issue, to ultimately give the site the polish and cohesion we were striving for.

Homepage

The homepage offered the biggest opportunities for solidifying our grid system. One of the biggest lessons learned from this portion of the site, was when working with large user submissions, keeping the layout simple is the best solution. From a technical standpoint and design standpoint, I think for a site like this, letting the work speak for itself and providing the container for it to shine, is more important than showing off trendy or new layouts made possible by CSS grid.

More generally, the homepage is the first page the user sees when navigating to the site. Featured images are shown from each student, and when you hover over the work, the students headshot and name is revealed in the side-navigation. We wanted to create a direct link between the work and the students headshot and name, creating stronger connections between the work and the designer.

Student Page

The student page was built using CSS grid and flexbox. Students submitted two headshots via the form and on hover the students image switches from classic to experimental. On smaller screens we disabled this functionality and displayed the classic headshots only. The students three areas of focus are also present under headshots, and the user is able to search and filter by student name and area of focus.

Single Student Page

The single student page is where the main content from the form submission lives. Each student was allocated three projects with different layout options for each project.
